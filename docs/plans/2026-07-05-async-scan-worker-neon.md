# Implementation Plan ‚Äî Asynchronous Scan via Worker-owned Neon Writes

Status: PROPOSED (not started)
Author: diagnosis follow-up, 2026-07-05
Related incident: scans failing with "The operation was aborted due to timeout"
(Vercel‚Üíworker dispatch fetch aborted at 270s; worker never streamed).

## 1. Problem this milestone removes

Today a full Gmail scan depends on ONE long-lived synchronous streaming HTTP
request from the Vercel `after()` callback to the Python worker's `POST /scan`.
The worker holds all progress and results *inside that stream*; it does not
persist anything itself. Consequences:

- If the stream stalls (cold Render worker, network blip, Render restart), the
  scan fails and any partial work is lost.
- The scan is bounded by Vercel's `maxDuration` (300s) and the 270s dispatch
  timeout, not by how long the work actually takes.
- Progress/results only reach Neon via the Vercel process relaying stream lines.

Target end state (matches the product requirement "a full Gmail scan must never
depend on a long-lived synchronous Vercel request"):

- Vercel creates the scan row, fires a short "start" request to the worker, and
  returns 202 immediately.
- The worker processes the scan independently and writes its OWN progress,
  status, and results directly to Neon.
- The UI polls Neon (`GET /api/scans/{id}/progress`) exactly as it does now.
- A stalled dispatch cannot lose or strand a scan.

## 2. Scope / non-goals

In scope: scan lifecycle (create ‚Üí dispatch ‚Üí process ‚Üí persist ‚Üí complete),
worker DB access, idempotency, retries, stuck-scan recovery, deploy sequencing.

Out of scope (unchanged): classification pipeline, export/screenshot paths,
OAuth/connection storage, supplier reconciliation logic. These keep working as
they do; only WHERE scan progress/results are written moves.

## 3. Schema changes (Prisma + Neon)

Additive only ‚Äî no destructive migration; deploy with `db push` per repo
convention (never `migrate dev`).

`Scan` model additions:

| Column | Type | Purpose |
|---|---|---|
| `dispatchId` | `String? @unique` | Idempotency key for a single processing attempt. Generated by the web app at create time; the worker refuses to start a second run for the same `dispatchId`. |
| `workerStartedAt` | `DateTime?` | Set by the worker when it accepts the job (distinct from `startedAt`, which the web sets on enqueue). Distinguishes "accepted by web" from "picked up by worker". |
| `heartbeatAt` | `DateTime?` | Worker touches this every N seconds while processing. Stuck-scan recovery keys on it. |
| `attempt` | `Int @default(0)` | Increments per retry; caps runaway reprocessing. |
| `workerVersion` | `String?` | Worker writes its `WORKER_VERSION` so a stuck/failed scan can be traced to a build. |

Existing `progress`, `progressMessage`, `status`, `totalMessages`,
`processedCount`, `invoiceCount`, `errorMessage`, `completedAt` are now written
by the WORKER instead of the web relay. `updatedAt` (@updatedAt) remains the
staleness signal.

Keep the `one_active_scan_per_org` partial unique index as-is ‚Äî still the
authoritative duplicate-scan guard.

Add index on `(status, heartbeatAt)` to make the recovery sweep cheap.

## 4. Worker database access boundaries

Principle: the worker gets the NARROWEST possible database capability.

- Connection: a dedicated Neon role `scan_worker` with a separate
  `DATABASE_URL_WORKER` secret (set in Render env only, never in the web app).
- Grants: `SELECT, INSERT, UPDATE` on `scans` and `invoices` only. No `DELETE`,
  no DDL, no access to `users`, `accounts`, `sessions`, `gmail_connections`,
  `organizations`, `exports`, `suppliers`. (The worker never needs to read
  OAuth tokens from the DB ‚Äî the web app continues to pass the access/refresh
  token in the start request body, so tokens never live in a worker-readable
  table.)
- Row scoping: every worker write is constrained by `id = :scanId AND
  organizationId = :orgId` (both passed in the start request, signed by the
  worker bearer secret). The worker never writes a scan it was not handed.
- Use a small, pinned Postgres driver (`asyncpg`/`psycopg`) with explicit
  statement timeouts; no ORM needed for ~5 statements.
- Pooling: use Neon's pooled endpoint (PgBouncer) for the worker to avoid
  connection storms during concurrent scans.

Terminal-state guard MUST be replicated in the worker's UPDATEs (mirror
`updateScanStatus`): `WHERE status NOT IN ('COMPLETED','FAILED','CANCELLED')`
so a late worker write can't resurrect a cancelled scan.

## 5. Idempotency

- Web generates `dispatchId` (cuid) on scan create and includes it in the
  start request.
- Worker, on receipt: `UPDATE scans SET status='RUNNING', workerStartedAt=now(),
  workerVersion=:v WHERE id=:scanId AND dispatchId=:dispatchId AND status IN
  ('PENDING','RUNNING') AND workerStartedAt IS NULL RETURNING id`. Zero rows ‚Üí
  already picked up (or cancelled) ‚Üí worker returns `{accepted:false, reason}`
  and does NOT process. This makes a duplicate start request a no-op.
- All progress writes are last-write-wins on `(progress, progressMessage)`;
  they carry no side effects, so at-least-once delivery is safe.
- `bulkCreateInvoices` already uses `createMany(skipDuplicates)` keyed on
  `(organizationId, gmailMessageId)` ‚Äî invoice persistence is already
  idempotent and can be re-run safely on retry.

## 6. Retries

- Web‚Üíworker START request is small and fast (no streaming). Retry it up to 3√ó
  with short backoff; because start is idempotent (Section 5), a retry after a
  network timeout cannot double-process.
- If all start attempts fail, the scan stays PENDING and the recovery sweep
  (Section 7) will either re-dispatch or fail it ‚Äî the scan is never silently
  lost.
- Worker-internal Gmail calls keep their existing `_exec` backoff
  (429/5xx, 3 retries). Unchanged.
- Add a bounded re-dispatch: a PENDING scan with no `workerStartedAt` after T
  seconds is re-sent once (increment `attempt`); `attempt >= 2` ‚Üí FAILED with a
  precise message. Prevents infinite redelivery.

## 7. Stuck-scan recovery

Replace the current "relay stopped writing" heuristic with worker heartbeats:

- Worker touches `heartbeatAt` every ~5s while processing.
- Recovery sweep (cron + opportunistic on POST /api/scans, as today) marks a
  scan FAILED when: `status='RUNNING' AND heartbeatAt < now() - STALE` (e.g.
  90s), OR `status='PENDING' AND createdAt < now() - START_DEADLINE` with no
  `workerStartedAt` (worker never accepted). Keep the existing `startedAt >
  15min` hard fallback.
- Because the worker owns the terminal write, recovery only ever flips
  genuinely-dead scans; a slow-but-alive worker keeps its heartbeat fresh and
  is never reaped.
- Recovery message must be precise (name the phase: "worker never accepted"
  vs "worker stopped heart-beating"), reusing the `sanitizeScanError` style.

## 8. Deployment sequencing (safe, reversible, no big-bang)

1. **Schema first (backward-compatible).** `db push` the additive columns.
   Old code ignores them. No behavior change. Verify tree/tests green.
2. **Worker DB role + secret.** Create `scan_worker` Neon role with the grants
   in Section 4; set `DATABASE_URL_WORKER` in Render only. Deploy a worker that
   OPENS the connection and writes heartbeats/progress **in addition to**
   streaming (dual-write), still returning the stream. Web still reads the
   stream ‚Äî no dependency on DB writes yet. This proves worker DB access in
   production with zero risk.
3. **Flip the reader.** Change the web `after()` to fire a non-streaming START
   request and return; stop consuming the stream. UI already polls Neon.
   Gate behind an env flag (`ASYNC_SCAN=1`) so it can be toggled off instantly.
4. **Recovery swap.** Move the sweep to heartbeat-based (Section 7) behind the
   same flag.
5. **Remove streaming.** Once stable for N days, delete the stream-relay path
   and the flag. Worker returns `202 {accepted}` only.

Rollback at any step = unset `ASYNC_SCAN` (steps 3-4) or redeploy previous
worker (step 2). No destructive schema change to undo.

## 9. Testing

- Unit: worker start-idempotency (0-row UPDATE ‚Üí not accepted); terminal-guard
  on worker writes; recovery predicate with heartbeat staleness.
- Integration: kill the worker mid-scan ‚Üí recovery marks FAILED with a precise
  message, no partial COMPLETED. Duplicate START ‚Üí single processing run.
- Keep the existing `scan-timeouts.test.ts` chain guard; add a heartbeat-cutoff
  ordering assertion.

## 10. Risks / open questions

- Neon connection budget under concurrent scans (mitigate with pooled endpoint
  + low worker concurrency).
- Render free tier still cold-starts; async removes the 270s failure but the
  FIRST scan after idle is still slow until the external keep-warm monitor
  (separate, already recommended) is in place.
- Worker now needs DB creds ‚Äî expands the worker's blast radius; the narrow
  role in Section 4 is the primary control. Audit that the worker image/logs
  never print `DATABASE_URL_WORKER`.
